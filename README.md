転売プロジェクト（resale-arbitrage）の**小額モデル検証**のためのツール。
`docs/07_minimal_validation.md` / `docs/08_broad_screening.md` の **Q1「入手再現性」** を
実測するために、狙いの商品が正規店で定価・在庫ありになった瞬間を検知して即通知する。

> ⚠️ これは利益を取りに行く投資ではなく、「このモデルが回るか（定価で争奪戦に勝てるか）」を
> 最小コストで見極める実験のための入手監視Botです。

## 🔄 動的追従システム（2026-06-22 / docs/11_dynamic_strategy.md）
固定リスト監視から「市場の移り変わりに自動追従する3層システム」に進化させた。

**Phase1（地ならし・新弾検知）:**
- `compute_page_signature` に揮発日付/揮発行の除外を追加 → 既存page_updateの誤検知バグを一掃
- ポケカ公式API新弾検知（`pokecard_official_list`）: resultAPI.php 4カテゴリの (商品名,発売日) 差分で新商品を自動検知
- ポケセン店頭ニュース(tokyodx/megatokyo)・公式info一覧を監視（都内店頭・コラボ/限定）

**Phase2（発見層）:**
- `discover_from_rss()`: nyuka-now instockフィードから監視キーワード(ポケカ/ワンピ/遊戯王/DBFW/コナン)
  に合致する入荷・再販商品を自動発見 → 固定リストを動的に補完。link差分で新規のみ通知。

**Phase3（相場選別・現在ログ通知モード）:**
- `fetch_altema_box_prices()` + `passes_profit()`: altema BOX相場で「market/retail」を判定。
  ポケカは別格で閾値1.0（定価以上で売れれば監視）。定価割れ銘柄は「除外候補」として警告。
- ⚠️ 現在は **ログ通知のみ**（`AUTO_DROP_ENABLED=False`）。数日様子を見て誤除外がないことを
  確認してから自動除外を有効化する安全運用。

> **ポケカは別格**: ポケカ関連は新弾・過去BOX・コラボ・グッズを広く拾い、相場閾値も緩い。

## ☁️ 実行体制（2026-07-03 クラウド主戦・ローカルは平日ボーナス層）

2026-07-02〜03 の総点検で判明した事実と対策:

1. **nyuka-now がクラウドIPを遮断**: Actionsランナーからは接続タイムアウトになり、
   nyuka-now系の監視11件＋RSS発見器が**全滅**していた（ローカルMacからは0.5秒で取得可能）
   → **主要銘柄は anime-matsuri の「抽選予約・先着販売・再販入荷まとめ」ページに移行**
     （同種の集約情報・クラウド到達可・OP-16ページで監視実績あり）。
     anime-matsuriに専用ページが無い銘柄（DBFW SB01/FB10・遊戯王・コナン）のnyuka-now監視は
     ローカル実行時のみ機能するボーナス層として残置（サーキットブレーカーでクラウド時のコスト極小）
2. **無料枠切れ**: privateリポジトリのActions無料枠は月2,000分。旧設定（1回約28分×日10回）は
   月初8日で枠が尽き、**残り3週間は全停止**していた（6/30の連続failureの正体）
   → サーキットブレーカーで1回2〜3分に短縮し、**毎時1回×1パス**（月約1,500〜1,900分）で枠内
3. **cron間引き**: 高頻度指定ほど間引かれる。毎時指定でも遅延はあり得る前提で運用
4. Macは土日祝に起動しないため、ローカルlaunchdは主戦にできない
   → 平日日中だけ10分間隔の密な監視を上乗せするオプション（`bash local/setup_local.sh`）

**さらに強化する2つの選択肢（要判断）:**
- リポジトリをpublic化 → Actions分数が無料無制限になり15分間隔に引き上げ可能
- 楽天市場API（無料のアプリID登録）→ 楽天ブックスの定価在庫を公式APIで直接監視可能

- ローカルとActionsは状態ファイルが別なので、同じイベントを両方が拾うと通知が最大2回来る
  （見逃しゼロを優先した仕様）

## 🎯 機能

- ✅ 複数サイト・複数判定方式に対応（`method` で切替）。商品ページを監視し在庫復活を検知
- ✅ 「在庫なし→在庫あり」に変化した時だけ通知（誤通知・連投なし）
- ✅ **差分つき告知更新通知（2026-07-02）**: page_update の通知に「新規に現れた行」を最大6行
  そのまま載せる。従来の「更新されました（リンク先で確認）」から、
  「更新検知・新規2行: 7/5 10:00 Amazon予約受付開始 ／ …」のように**何が起きたかが読める**通知へ
- ✅ 件名に商品名を表示（「N件」だけの件名を廃止。開かなくても対象がわかる）
- ✅ RSS発見器の精度向上: ポケカ以外はサプライ用品（スリーブ/プレイマット等）を除外
- ✅ 接続不能ホストのサーキットブレーカー: 同一パス内で再試行せず即スキップ
  （nyuka-nowタイムアウト連発で1起動28分かかっていた問題を解消）
- ✅ メール＋Discord(Webhook)の二重通知
- ✅ ローカルMac(launchd)で10分ごと自動実行＋GitHub Actionsが保険（3時間ごと）
- ✅ 取得失敗時は前回状態を維持（誤通知防止）

## 📊 データ源と在庫判定（事前検証で確定）

### 判定方式（method）
- **gdb_soldout**（ガンプラ）: GunplaDatabase の `shop_status_container` ブロックの
  `soldout` クラス/「売切」テキストの有無で店舗別在庫を判定。
- **toei_stock_status**（OP-16等）: 東映アニメ公式の商品ページ埋め込みJSONの
  `stock_status` を読む。`"0"` なら在庫なし、それ以外なら在庫あり。Shift_JIS。

### なぜ GunplaDatabase なのか
- **ヨドバシ.com 直接監視は不可**: Bot対策が強く、自動取得（curl/WebFetch）が接続失敗・タイムアウト。
- **GunplaDatabase は取得可能**で、複数店舗（Amazon等）の在庫状態を集約している。
- → GunplaDatabase の商品個別ページ（`?no=2294`）を監視する。

### 在庫判定ロジック
ページ内の `shop_status_container` ブロックごとに、CSSクラス `soldout` または
「売切」テキストの有無で店舗別の在庫を判定する。
**soldout でない店舗が1つでもあれば「在庫あり」** とし、前回「在庫なし」からの変化を通知する。

### ⚠️ 重要な注意（検証の前提）
- 検知される在庫には **Amazon等の転売価格**が含まれる場合がある。
- **通知が来たら「定価7,700円で買えるか」を必ず人間が確認する**こと（通知文に明記）。
- 定価で買えなければ見送る（＝Q1「定価入手の再現性」の検証にならないため）。

## 🛠️ セットアップ

### ローカル実行
```bash
pip install -r requirements.txt
python check_stock.py
```
※ メール/Discord通知には環境変数が必要。未設定なら検知のみ（送信スキップ）。

## 🔔 通知が来たら（検証の手順）

1. **まず価格を確認** — 定価7,700円で買えるか？（Amazon等は転売価格の場合あり）
2. 定価なら**即購入手続き**（数分で完売の争奪戦）
3. **記録**（docs/07 のKPI）: 何回目の挑戦で買えたか・所要時間・売り切れまでの分数
4. 確保できたらフリマ出品 → 売却日数・実際の手残りを記録

## 📝 監視対象の追加・変更

`config.py` の `WATCH_ITEMS` に GunplaDatabase の商品ID（`no=`）を追加する。
※対象は方針に合うもの（非酒類・非TCG・正規新品・未開封のまま売れる）に限ること。
